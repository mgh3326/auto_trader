# DATA KNOWLEDGE BASE

## OVERVIEW
`data/` is a mixed boundary: maintained Python loader modules plus static reference assets.

## WHERE TO LOOK
| Task | Location | Notes |
|------|----------|-------|
| KR/US stock code loaders | `data/stocks_info/*.py` | Master-code parsing/loading and lazy constants |
| Coin pair loader | `data/coins_info/upbit_pairs.py` | Upbit market/pair loading and cache behavior |
| Disclosure index loader | `data/disclosures/dart_corp_index.py` | DART corp index fetch/cache path |
| Package exports and prime helpers | `data/stocks_info/__init__.py`, `data/coins_info/__init__.py` | Public loader utility surface |
| Static reference headers | `data/stocks_info/*.h` | Vendor/reference schema files (non-runtime python) |
| Generated bytecode artifacts | `data/**/__pycache__/` | Generated files, not maintained source |

## CONVENTIONS
- Keep runtime data-loading logic in Python modules (`get_*`, `prime_*` patterns).
- Treat `.h` files under `data/stocks_info/` as static/reference assets.
- Keep package exports in `__init__.py` aligned with actual loader ownership.
- Prefer targeted loader updates over broad directory-wide changes.

## ANTI-PATTERNS
- Do not treat `__pycache__` contents as source of truth.
- Do not mix generated/runtime cache artifacts into maintained source workflows.
- Do not rename or remove static header assets without verifying dependent parser modules.
- Do not duplicate loader logic across `stocks_info`, `coins_info`, and `disclosures` without clear boundary reasons.

## NOTES
- `data/stocks_info/` has the largest mixed-content surface and needs extra care during refactors.
- Root `pyproject.toml` excludes data directories from Ruff checks, so local validation may need explicit targeted checks.
