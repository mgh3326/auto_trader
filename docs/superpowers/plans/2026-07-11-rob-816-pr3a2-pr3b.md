# ROB-816 PR-3a-2 and PR-3b Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Complete loss-cut approvals from Telegram callbacks and route Toss proposals through Toss-native preview and placement without any live broker call in tests.

**Architecture:** The Telegram callback installs a configured, fail-closed caller identity only around click-time revalidation and submission, then restores the prior context. Revalidation keeps its broker-neutral state machine but its production binding dispatches `toss_live` proposals to a narrow Toss adapter that converts Decimal arguments, translates `payload_preview`, and normalizes accepted Toss responses into the existing ACK/resting contract.

**Tech Stack:** Python 3.13, asyncio, ContextVar, FastAPI settings, SQLAlchemy async services, pytest/pytest-asyncio, Ruff, ty.

## Global Constraints

- `ORDER_PROPOSALS_SUBMIT_AGENT_ID` has no identity default; missing configuration remains fail-closed for `loss_cut`.
- Operators must explicitly add the configured submit identity to `LOSS_CUT_ALLOWED_AGENT_IDS`.
- No UUID is hardcoded for the Telegram callback submit identity.
- Broker and Telegram calls are mocked; no live mutation is allowed.
- Toss receives only its public `str | int` numeric contract and never a float.
- Toss tick adjustment remains owned by `toss_preview_order`; proposal revalidation consumes the normalized `payload_preview` and never adjusts ticks itself.
- Pull request is opened but not merged.

```python
def _toss_decimal_arg(value: Decimal | None) -> str | int | None:
    if value is None:
        return None
    normalized = value.normalize()
    return int(normalized) if normalized == normalized.to_integral_value() else format(normalized, "f")
```

Preview calls `toss_preview_order` and translates `payload_preview`. Submit calls `toss_place_order(dry_run=False, confirm=True, approval_hash=..., rung=...)`, preserving its client order ID as the proposal idempotency key. Non-Toss calls remove `account_mode` before delegating to `_place_order_impl`.
