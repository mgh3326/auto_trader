## Decision 1 — Existing domain snapshot tables are referenced via `source_ref_*`, not re-dumped

### Current state (code-grounded)

`app/models/` 에 이미 8개의 domain-specific snapshot/read-model 테이블이 있습니다:

| File | Table | Owner service | Shape |
|---|---|---|---|
| `invest_screener_snapshot.py` | `invest_screener_snapshots` | `services/invest_screener_snapshots/` | typed columns + `closes_window` JSONB |
| `investor_flow_snapshot.py` | `investor_flow_snapshots` | `services/investor_flow_snapshots/` | typed flow columns |
| `market_quote_snapshot.py` | `market_quote_snapshots` | `services/market_quote_snapshots/` | quote columns |
| `market_valuation_snapshot.py` | `market_valuation_snapshots` | `services/market_valuation_snapshots/` | valuation columns |
| `invest_momentum_event_snapshot.py` | `invest_momentum_event_snapshots` | `services/invest_momentum_events/` | momentum columns |
| `crypto_insight_snapshot.py` | `crypto_insight_snapshots` | `services/crypto_insight_snapshots/` | crypto signals |
| `invest_crypto_screener_snapshot.py` | `invest_crypto_screener_snapshots` | `services/invest_crypto_screener_snapshots/` | crypto screener |
| (also `review.investment_reports.market_snapshot/portfolio_snapshot`) | embedded JSONB on `investment_reports` | `services/investment_reports/` | report-embedded JSONB |

각 테이블은 자체 repository (`InvestScreenerSnapshotsRepository.upsert` 등), freshness 컬럼 (`snapshot_date`, `computed_at`, `collected_at`), 그리고 idempotency 컨벤션이 이미 있습니다. ROB-204에서 추가된 `InvestScreenerSnapshotsRepository.upsert` 만 허용되는 write path 같은 안전 경계도 그대로 살아 있습니다.

### Decision (locked)

**Generic JSONB로 재-dump하지 않는다.** 신규 `review.investment_snapshots` 는 도메인 row를 **reference** 합니다.

도메인 row를 가리키는 reference 패턴은 **two-column URN style**:

```python
source_table: str  # ex: 'invest_screener_snapshots', 'review.investment_watch_alerts'
source_id: int      # 해당 테이블의 PK
source_uri: str     # ex: 'invest_screener_snapshots:42' (denormalized, indexed)
```

Rationale:
- payload 중복 방지 (메모리: clean-cut preference — transitional layer/이중 source 회피)
- 도메인 테이블의 upsert/typed-column 안전성을 그대로 활용
- bundle은 "어떤 row들이 이 리포트에 들어갔나" 의 reference graph로만 기능

**Exception — generic payload가 정당화되는 경우 (whitelisted):**
1. 도메인 row가 아직 없는 새 source — `naver_remote_debug`, `toss_remote_debug`, `browser_probe`, `invest_page_probe`. 이들은 v1에서 도메인 테이블을 새로 만들지 않고 `payload_json` 으로 받음.
2. AI 호출 입력의 **재현용 frozen view** — required source 들을 정확히 한 형태로 캐시. payload는 LLM 입력에 들어간 그대로의 bytes-stable form.

**Required vs whitelisted-generic 결정 trace:**

| Snapshot kind | Storage | Why |
|---|---|---|
| `portfolio` | source_ref → KIS account snapshot (정형 row 신설) | 정형 컬럼 가치 큼, 다른 surface도 곧 reuse |
| `journal` | source_ref → `review.trade_journals` (기존) | 이미 정형 |
| `watch_context` | source_ref → `review.investment_watch_alerts`/`_events` (기존, ROB-265) | 이미 정형 |
| `market` | source_ref → `market_quote_snapshots`/`market_valuation_snapshots` | 이미 정형 |
| `candidate_universe` | source_ref → `invest_screener_snapshots`/`invest_momentum_event_snapshots` | 이미 정형 |
| `symbol` | source_ref → `market_quote_snapshots`/지표 테이블 | 이미 정형 |
| `news` | source_ref → `market_events`/`research_reports` (ROB-128/140) | 이미 정형 |
| `naver_remote_debug` | generic `payload_json` (whitelisted) | 도메인 테이블 없음 |
| `toss_remote_debug` | generic `payload_json` (whitelisted) | 도메인 테이블 없음 |
| `browser_probe` | generic `payload_json` (whitelisted) | 도메인 테이블 없음 |
| `invest_page` | generic `payload_json` (whitelisted) | 도메인 테이블 없음 |
| `llm_input_frozen` | generic `payload_json` (재현용) | LLM 입력 bytes-stable freeze |

**Migration story for ROB-265 embedded snapshots:**
`review.investment_reports.market_snapshot` 와 `portfolio_snapshot` JSONB 컬럼은 v1에서 **유지**합니다 (legacy 호환). 신규 리포트는 빈 dict 로 저장하고 `snapshot_bundle_uuid` 만 채움. 기존 리포트는 두 컬럼이 그대로 보존되어 history 조회 가능. 컬럼 drop은 ROB-269 이후 별도 cleanup 이슈.

### 3b. `review.investment_snapshots`

```text
id                       BIGSERIAL PK
snapshot_uuid            UUID NOT NULL UNIQUE

snapshot_kind            TEXT NOT NULL
                         CHECK IN ('portfolio','market','news','symbol',
                                   'candidate_universe','browser_probe','invest_page',
                                   'journal','watch_context',
                                   'naver_remote_debug','toss_remote_debug',
                                   'llm_input_frozen')
market                   TEXT NOT NULL CHECK IN ('kr','us','crypto')
account_scope            TEXT NULL  -- 같은 enum as runs
symbol                   TEXT NULL  -- KR 6자리 / US ticker; CLAUDE.md의 dot-format 규약

-- ★ Decision 1: source_ref triple (URN style). Whitelisted-generic은 모두 NULL.
source_table             TEXT NULL

-- generic payload (whitelisted kinds 만 사용)
source_kind              TEXT NOT NULL
                         CHECK IN ('kis_mcp','auto_trader_mcp','invest_api',
                                   'naver_remote_debug','toss_remote_debug',
                                   'combined','news_ingestor','manual','domain_ref')
                         -- 'domain_ref' = source_ref_* 가 채워진 경우

source_timestamps_json   JSONB NOT NULL DEFAULT '{}'
coverage_json            JSONB NOT NULL DEFAULT '{}'

collected_at             TIMESTAMPTZ NOT NULL DEFAULT now()

freshness_status         TEXT NOT NULL
                         CHECK IN ('fresh','soft_stale','hard_stale','partial','unavailable')

-- ★ Decision 3: canonical hash. KIS quote 같은 sub-second timestamp 가 들어가는
--   payload는 hash 가 항상 달라져 dedup 못 함. 따라서 hash는 "정규화된" payload 에 대해.
canonical_payload_hash   TEXT NOT NULL
                         -- sha256(canonical_json(payload without volatile fields))
                         -- 정규화 규칙:
                         --   1) keys lexicographic sort
                         --   2) source_timestamps 제외
                         --   3) sub-second timestamps strip → second precision
                         --   4) float → 9 자리 고정 소수
                         -- 정규화 함수: app/services/action_report/common/canonicalize.py
idempotency_key          TEXT NOT NULL UNIQUE
                         -- 권장 형태: f"{run_uuid}:{snapshot_kind}:{symbol or '_'}:{canonical_payload_hash[:12]}"

created_at               TIMESTAMPTZ NOT NULL DEFAULT now()

UNIQUE (canonical_payload_hash, snapshot_kind, market, account_scope)
  -- dedup constraint: 같은 정규화 payload는 한 row만. 새 run에서 재사용 시
  --   bundle item만 새로 연결.
INDEX (snapshot_kind, market, symbol, as_of DESC)
INDEX (source_uri)
INDEX (run_id)

-- ★ Append-only invariant: UPDATE/DELETE 금지. 서비스 레이어에서만 보장
--   (DB trigger는 v1에 두지 않음 — 마이그레이션 rollback 비용 큼)
```

#### 3b-1. Dedup semantic — "first writer wins" (Phase 1 implementation 후 명시)

UNIQUE `(canonical_payload_hash, snapshot_kind, market, account_scope)` 는 의도적으로 `run_id` 를 포함하지 않습니다. 같은 payload가 다음 run 에서 또 들어오면 dedup 으로 **기존 row가 반환**되고, 새 row 는 만들지 않습니다. 결과적으로:

* 반환된 row 의 `run_id` / `idempotency_key` 는 **첫 writer** 의 것 — 현재 caller 의 것이 아님.
* 현재 run이 이 snapshot 을 "사용했다" 는 사실은 **`investment_snapshot_bundle_items` 의 link 가 단일 source of truth**. snapshot row 자체에 caller-run 정보를 쓰지 않음.
* `idempotency_key` 의 `run_uuid` 컴포넌트는 "이 snapshot 을 최초로 만든 run" 의 trace 일 뿐, "이 snapshot 을 참조하는 모든 run" 의 enumeration 이 아님.
* "이 snapshot 이 내 run 의 것" 인지 확인하려는 caller 는 `snapshot.run_id == my_run.id` 로 체크하고, `!=` 인 경우를 **정상 reuse** 로 취급해야 함 (에러로 처리하지 말 것).

이 결정은 cross-run dedup 의 본래 목적 (Phase 8 scheduler 가 5분마다 같은 portfolio 를 새로 dump 해도 row 가 폭증하지 않게) 과 정렬됩니다. 대안 (UNIQUE 에 `run_id` 포함) 은 dedup 을 run-local 로 만들어 row 수를 cross-run 마다 곱해버리므로 reject.

운영 시 caller 가 "내 run 의 idempotency_key 를 query 로 찾고 싶다" 는 use case 가 생기면 bundle_items + run join 으로 처리하고, idempotency_key 자체를 caller-run scoped 로 만들지 않습니다.

## Decision 4 — Stale gate is enforced at THREE layers: DB CHECK + generator + post-generation deterministic linter

### Why three layers

ROB-265 의 advisory-only invariant 는 `CHECK CONSTRAINT` (DB), `execution_mode = 'advisory_only'` 강제 (model), 그리고 service-level validation 으로 3중 방어 중. ROB-269 의 stale gate도 같은 패턴.

| Layer | Where | What it catches |
|---|---|---|
| (i) DB CHECK | Decision 3e `ck_investment_reports_no_published_on_hard_stale` | hard_stale/failed bundle 로 published 시도 |
| (ii) Generator gate | `action_report/common/snapshot_bundle.py::guard_action_language` | 리포트 생성 시점에 LLM 프롬프트로 "no executable action" 지시 + bundle status 기반 의사결정 분기 |
| (iii) Post-gen linter | `action_report/common/stale_gate.py::lint_action_language` | LLM 출력에 매수/매도 단어 들어가면 reject — deterministic regex |

LLM 은 안 듣는 경우가 있으므로 (ii) 만으로는 약합니다. (iii) 가 안전 net.

### (iii) Post-generation linter — TDD outline

`app/services/action_report/common/stale_gate.py`:

```python
ACTION_VERBS_KR = {"매수", "매도", "사세요", "파세요", "추격", "분할매수", "분할매도", "익절", "손절"}
ACTION_VERBS_EN = {"buy", "sell", "long", "short", "add", "trim", "stop"}
SOFT_VERBS_KR = {"관망", "보유 유지", "지켜보기", "확인 불가"}

@dataclass(frozen=True)
class StaleLintResult:
    ok: bool
    violations: list[StaleLintViolation]  # snapshot_kind, text excerpt, matched_verb

def lint_action_language(
    *,
    report_text: str,
    bundle_status: str,                       # 'complete' | 'partial' | 'stale_fallback' | 'failed'
    freshness_summary: Mapping[str, Any],
    account_scope: str | None,
) -> StaleLintResult:
    """If bundle is stale/account snapshot unavailable, action verbs are forbidden."""
```

Test cases (Phase 5의 첫 commit에 들어감, 구현 전 작성):

| Test | bundle_status | account scope | report_text | Expected |
|---|---|---|---|---|
| `test_lint_passes_on_fresh_complete` | complete | kis_live | "삼성전자 매수 검토" | ok=True |

## Sub-decision defaults locked (앞 응답의 open question 3개 처리)

광현님이 "Phase 1 구현으로 들어가면 된다" 고 재확인했으므로, 이전 응답에서 남겨둔 3개 open question 을 다음 default 로 락합니다. v2 에서 뒤집기 쉬운 결정만 default 처리:

1. **`naver_remote_debug` / `toss_remote_debug` 도메인 테이블** — **v1은 whitelist generic 유지**. 이유: (a) 이 두 source 가 reviewer/리포트에 자주 cite 되는 패턴이 아직 데이터로 확인되지 않음, (b) 도메인화는 reviewer/리포트가 이들을 typed query 로 자주 조회할 때 가치가 생기는데, v1에서는 reference + payload preview 만으로 충분, (c) 도메인 테이블 신설은 또 다른 migration + repository + service tier 를 의미해 Phase 1 PR 크기를 키움. 향후 reviewer query 패턴이 정착되면 v2 cleanup 으로 정형화.

2. **`requested_by` enum 5개 (`hermes`, `user`, `scheduler`, `claude_code`, `reviewer`)** — **5개로 락**. reviewer agent 의 세부 종류 (codex/gemini/claude review 등) 는 enum 에 더하지 않고 `metadata.reviewer_kind` 로 표현. 이유: enum CHECK 는 마이그레이션이 필요하지만 metadata 는 free-form 이라 reviewer agent 가 추가될 때마다 schema 건드리지 않음.

3. **`canonical_payload_hash` sub-second strip 단위** — **single second precision 으로 통일**. snapshot_kind 별 가변 정책은 두지 않음. 이유: (a) quote 스냅샷이 들어오는 단위가 이미 second-precision 인 경우가 많아 추가 dedup 이점이 작음, (b) snapshot_kind 별 정규화 분기는 canonicalize 함수의 분기를 늘려 hash collision 디버깅 비용을 키움, (c) minute precision 까지 strip 해야 의미 있는 case 는 분봉 dedup 뿐인데, 분봉은 어차피 `idempotency_key` 의 `as_of` 컴포넌트로 dedup 되도록 service tier 에서 처리.

위 3개 default 가 운영 중 잘못된 결정으로 드러나면, 모두 **additive 변경**으로 뒤집을 수 있음 (enum 확장, hash 정규화 함수 교체, 도메인 테이블 신설). v1에서 conservative 방향으로 락하고 v2에서 evidence 기반 확장.
