# ROB-816 PR-3 Design

## Goal

Finish the follow-up approval workflow in three independently reviewable pull
requests based on `main` at or after `6e5795e2`. Preserve the existing
`order_proposals` ledger as the single source of truth while fixing activation
smoke findings, routing Toss orders through the Toss-specific approval binding,
and projecting broker-confirmed fills back into proposal rungs.

The quality-contract Markdown migration remains out of scope.

## Shared constraints

- Each PR starts independently from `main`; none is stacked on another PR.
- Broker and Telegram calls are mocked in tests. No live mutation is allowed.
- Changes use test-driven development and each PR must pass its focused tests
  and `make lint` before opening.
- Pull requests are opened but never merged by this work.
- Database changes are additive only.

## PR-3a: approval UX and safety

### Guard versus internal-error classification

Revalidation keeps the existing fail-closed behavior but stops treating every
`success: false` preview as a policy guard. A `guard_blocked` outcome requires
both `success: false` and a recognized guard error. An unrecognized failure
envelope, including a wrapped implementation exception, returns the existing
`error` outcome. Both paths return the rung to `pending_approval`; neither sends
an order.

The Telegram result summary renders a bounded, Markdown-escaped summary of
`outcome.detail.error` for rejected, blocked, and error outcomes. Operators see
the actual reason instead of only a generic label, without exposing approval
tokens or other secret-bearing fields.

### Proposal expiry and operator void

`OrderProposalsService.create_proposal` owns the default validity rule so every
caller receives it: absent `valid_until` becomes the next `00:00 KST` boundary
(the end of the proposal's KST creation day). Explicit values must be timezone
aware and later than creation time.

Approval checks expiry while holding the proposal row lock and before recording
approval or submitting. An expired proposal moves every eligible pre-submit
rung to `expired`, invalidates the outstanding approval nonce, commits that
state, and reports that the proposal expired. It cannot be approved later.

`order_proposal_void(proposal_id, reason)` is an operator MCP tool. The reason
must contain non-whitespace text. It voids only rungs whose state proves no
broker mutation can be outstanding (`draft`, `pending_approval`,
`revalidating`, `needs_reconfirm`, or `approved`), records the group reason,
sets `no_resubmit`, and invalidates the nonce. It refuses ambiguous or
post-submit rungs such as `submitting`, `unverified`, `acked`, `resting`, and
`partially_filled` rather than erasing possible broker state.

### Loss-cut proposal binding

The following nullable columns are added to `review.order_proposals`, not to
individual rungs:

- `exit_intent TEXT NULL`
- `exit_reason TEXT NULL`
- `retrospective_id BIGINT NULL`
- `approval_issue_id TEXT NULL`

`NULL` `exit_intent` is the backward-compatible ordinary-order path. The only
accepted non-null value is currently `loss_cut`. These fields are validated and
stored directly; `rationale` JSON is never parsed to recover them.

At create time, `exit_intent="loss_cut"` requires all other three fields. The
service performs the retrospective checks that are stable at proposal creation:
the row exists, the symbol matches, the trigger type is `stop_loss` or
`thesis_change`, and it is no more than 72 hours old. The service also requires
a live KIS equity sell limit proposal and a structured `exit_reason` accepted by
ROB-800. This is an operator UX validation layer, not the final execution gate.

Paperclip `approval_issue_id` status is deliberately not checked at create
time. The issue may change between proposal creation and the Telegram click.
Both click-time preview and submit pass `exit_intent`, `exit_reason`,
`retrospective_id`, and `approval_issue_id` unchanged to `_place_order_impl`.
The existing ROB-800 validator therefore checks current Paperclip `done` state,
caller authorization, retrospective validity, price band, and approval binding
at the moment of approval and again at submit.

The Telegram approval message displays `exit_reason` and `retrospective_id` for
loss-cut proposals. The four fields are also included in proposal read/list
responses and the immutable proposal payload hash so replacement and approval
audit cannot silently change the exit intent.

## PR-3b: Toss routing adapter

The service allowlist adds `("toss_live", "equity_kr")` and
`("toss_live", "equity_us")`. This does not open the shared
`_place_order_impl` path.

The production revalidation binding dispatches by `account_mode`:

- KIS and Upbit retain `_place_order_impl`.
- Toss preview calls `toss_preview_order` with market `kr` or `us`.
- Toss submit calls `toss_place_order` with `dry_run=False`, `confirm=True`,
  the preview's `approval_hash`, and the same rung discriminator.

A dedicated decimal adapter converts proposal `Decimal` values only to the
Toss public contract's `str | int` types: integral values become `int` when
safe for the parameter, and fractional values use canonical decimal strings.
No float conversion occurs on the Toss path.

The Toss preview adapter reads normalized price, quantity, and client order ID
from `payload_preview`; the submit adapter translates Toss's accepted response
into the existing proposal ACK/resting result contract while preserving
`broker_order_id`, `correlation_id`, idempotency key, and approval digest.
Revalidation tests pin both the route choice and the exact approval-hash/rung
handoff.

## PR-3c: reconcile fill projection

Broker-confirmed cumulative fill evidence from all proposal-capable live lanes
is projected to `OrderProposalsService.record_fill_evidence`:

- `kis_live_reconcile_orders` for Korean KIS live orders;
- `live_reconcile_orders` for US KIS and Upbit orders;
- `toss_reconcile_orders` for Toss Korean and US orders.

Dry-run reconcile never writes proposal state. Non-dry-run projection uses both
the correlation ID and broker order ID when available, passes cumulative filled
quantity, and chooses `partially_filled` or `filled` from the broker verdict.
Projection happens immediately after broker evidence classification, before
later journal/ledger bookkeeping, because the proposal state is a projection of
broker truth rather than a claim that all downstream accounting succeeded.

`record_fill_evidence` remains idempotent despite the repository query having no
state filter. If the matched rung is terminal, it returns that rung unchanged.
If a later cumulative partial arrives while the rung is already
`partially_filled`, it updates `filled_qty` without attempting the illegal
`partially_filled -> partially_filled` transition. A final cumulative fill then
uses the legal `partially_filled -> filled` transition.
