# Upbit Shadow-Sim (crypto paper broker) — Design

Generated by brainstorming on 2026-07-04
Linear: ROB-703 (parent ROB-644, follow-up to ROB-697)
Branch: feature/ROB-703-upbit-shadow-sim
Status: DRAFT (revised: extend existing paper_trading, not a new ledger)

## Problem Statement

The operator wants to validate that **meaningful crypto analysis can drive buy/sell decisions executed on a mock account** — interactively (operator + Claude feedback loop), on the currently-open crypto market. Upbit has no paper API and `binance_demo` is a different exchange/currency, so we need an **Upbit-faithful paper account** (KRW, real Upbit symbols) whose fills track **live Upbit market data**, so analysis and execution are on the same venue. Automation/scheduling is deferred until the loop is proven.

### Why extend the existing paper-trading system (not build new)

Grounding found `app/services/paper_trading_service.py` (`PaperTradingService`) + `app/models/paper_trading.py` (`PaperAccount`/`PaperPosition`/`PaperTrade`/`PaperDailySnapshot`) **already provide a full crypto paper account/order/position/PnL/analytics simulator** that prices crypto via the live Upbit client (`fetch_multiple_current_prices`), applies the crypto fee (`FEE_RATES["crypto"]` = 0.05%), and is already exposed via MCP tools on the DEFAULT profile (`create_paper_account`, `reset_paper_account`, `list_paper_accounts`, `get_paper_performance`, `get_paper_trade_log`, `compare_paper_accounts`, + the paper order/portfolio handlers). Building a new `upbit_shadow` ledger would duplicate all of it (DRY violation).

**The one real gap** (verified): a limit order fills **instantly at the limit price** — `preview_order`/`execute_order` set `fill_price = Decimal(str(price))` for `order_type == "limit"` (paper_trading_service.py:220-223), with no check that the market actually traded there. That is unrealistic (you always get filled at your limit). The Upbit-faithful behavior the operator wants is a **resting limit order** that fills only when live Upbit price actually crosses it.

So this project is an **additive extension**: add realistic **resting limit orders + a reconcile fill engine**, reusing everything else.

## Scope

**v1 (this spec):**
1. A `paper_pending_order` table for resting limit orders (small, additive).
2. A **pure fill engine** deciding whether live Upbit OHLCV/trades crossed a resting limit since it was placed.
3. Service methods on/beside `PaperTradingService`: `place_limit_order` (resting; reserves buy cash), `reconcile_pending_orders` (fill crossed limits → book via the existing execute path), `cancel_pending_order`, `list_pending_orders`.
4. MCP tools for the above (mirror the existing paper tool pattern), on DEFAULT.
5. Tick-snap (reuse `adjust_price_to_upbit_unit`, floor variant) + min-order 5,000 KRW guard on limit placement.

**Preserved (unchanged):** existing market orders (`execute_order(market)` fills at live price) and the existing instant-limit `execute_order(limit)` behavior (backward-compat; the operator loop uses the new resting path instead). All existing paper tools, accounts, positions, PnL, analytics.

**Out of scope (fast-follow):** background fill engine (v1 reconcile is on-demand), partial fills, slippage/orderbook-depth fills, auto retrospective/forecast wiring, scheduling/headless automation.

## Architecture

Additive to the existing paper subsystem. New pieces are bounded:

```

           paper_cancel_pending_order / paper_list_pending_orders)
        │  (thin handlers → service)
        ▼
```

## Components

### 1. `paper_pending_orders` table (NEW, additive)

`app/models/paper_trading.py` — add `PaperPendingOrder` (register in `app/models/__init__.py` so `create_all` builds it in the test DB, exactly like the other paper models). Columns:
- `id` BigInteger PK; `account_id` FK → `paper_accounts.id`
- `symbol` (String, e.g. `KRW-BTC`), `side` (`buy`/`sell`), `order_type` (`limit` — only limits rest)
- `limit_price` Numeric(20,8), `quantity` Numeric(20,8) (resolved at placement from qty or amount)
- `reserved_krw` Numeric(20,4) (buy: gross+fee reserved from cash at placement)
- `status` (`pending`/`filled`/`cancelled` — CHECK)
- `thesis` (Text, nullable), `placed_at`/`filled_at`/`cancelled_at` (timestamptz), `fill_price` Numeric(20,8, nullable), `paper_trade_id` (FK → paper_trades.id, nullable — the booked fill)
- `created_at`/`updated_at` server_default now

### 2. Fill engine — pure functions (`app/services/paper_fills.py`, NEW)

- `snap_limit_down(price: Decimal) -> Decimal` — floor to the Upbit KRW tick band (derive floor from the same band thresholds as `adjust_price_to_upbit_unit`, using `Decimal` + floor, not round).
- `limit_crossed(side: str, limit_price: Decimal, bars: list[Candle]) -> Decimal | None` — buy fills if any bar `low <= limit_price`; sell fills if any bar `high >= limit_price`; returns `limit_price` on fill else `None`. No partial fills (all-or-nothing). Pure (Decimal in → Decimal/None out), no I/O; exhaustively unit-tested with canned `Candle`s.

Constants reused (do NOT re-hardcode): `FEE_RATES["crypto"]` + `calculate_fee` (paper_trading_service.py), min-order `5000` (`app/mcp_server/tooling/shared.DEFAULT_MINIMUM_VALUES["crypto"]`).

### 3. `PaperLimitOrderService` (NEW; reuses PaperTradingService)

`app/services/paper_limit_order_service.py`. Constructor takes an `AsyncSession` and composes a `PaperTradingService`. Methods:
- `place_limit_order(*, account_id, symbol, side, limit_price, quantity=None, amount=None, thesis=None)` — validate (tick-snap the price, min-notional 5,000 KRW, one-of qty/amount, sufficient cash for buy / position for sell). Insert a `pending` `PaperPendingOrder`; for a buy, reserve `reserved_krw = gross+fee` from `account.cash_krw` (so orderable cash reflects resting buys). No fill.
- `reconcile_pending_orders(*, account_id, now)` — for each `pending` order: fetch live Upbit OHLCV since `placed_at` via `market_data.service.get_ohlcv(symbol, "crypto", period, count, end=None)` (page/over-fetch since Upbit has no `since`); call `limit_crossed(...)`; on fill → release the reservation and book the trade through the EXISTING `PaperTradingService.execute_order(..., order_type="market", price=fill_price...)` fill-booking path (or a shared internal booker) so position/cash/PnL update consistently, set the order `filled` + link `paper_trade_id`. On no-cross → leave `pending`. On no live data → leave `pending` + note `data_unavailable` (never fabricate).
- `cancel_pending_order(*, account_id, order_id)` — release reservation, set `cancelled`.
- `list_pending_orders(*, account_id)` — pending orders + current live distance-to-fill.

### 4. MCP tools (NEW), on DEFAULT

`app/mcp_server/tooling/paper_limit_order_handler.py` (mirror the existing paper handler + kiwoom_mock dry_run/confirm gate). Tools: `paper_place_limit_order` (dry_run default + confirm), `paper_reconcile_orders`, `paper_cancel_pending_order`, `paper_list_pending_orders`. Export a `PAPER_LIMIT_ORDER_TOOL_NAMES: set[str]`; register on DEFAULT in `registry.py` next to `register_paper_account_tools` (line ~238); add the name set to `tests/test_mcp_profiles.py` `_ALL_ORDER_TOOL_NAMES` AND `_ORDER_SURFACE_MATRIX[McpProfile.DEFAULT]` (set-equality guard).

Deposit/reset/positions/cash/market-orders/analytics: use the EXISTING paper tools (`create_paper_account` for deposit, `reset_paper_account`, the existing order/portfolio handlers).

## Data flow (interactive loop — operational, no new build)

`create_paper_account(name, initial_capital_krw=…)` (입금) → Claude runs `route_request(intent, market="crypto")` live crypto analysis → proposes buy/sell w/ levels → operator review → `paper_place_limit_order(confirm=True)` (resting) or existing market order → later `paper_reconcile_orders` → existing `get_positions`/portfolio for PnL → interactive retrospect → iterate.

## Safety

Pure simulation — touches no real broker/Upbit (reads live market data + writes only paper tables). No real order/mutation surface reachable. `dry_run` default + `confirm` on limit placement for procedural consistency. Loss-sell guard NOT enforced (practice account; existing paper system already doesn't gate stop-loss). ROB-501: deterministic code, no LLM import.

## Error handling

- `place_limit_order` rejects: sub-tick price (auto snap-down, echo the snapped price), below 5,000 KRW, both/neither qty/amount, insufficient cash (buy)/position (sell), inactive/unknown account → typed error, no write.
- `reconcile` with no live OHLCV: leave `pending`, return `data_unavailable` (no fabricated fill).
- Idempotent reconcile: a `filled`/`cancelled` order is never re-processed.

## Testing

- **paper_fills.py:** pure unit tests (`@pytest.mark.unit`, no DB, mirror `tests/services/brokers/kis/test_live_order_expiry.py`): buy fills when a bar low ≤ limit; sell fills when high ≥ limit; no-cross → None; tick floor-snap bands. Run: `uv run --all-groups pytest tests/services/test_paper_fills.py -v --no-cov`.
- **PaperLimitOrderService:** integration tests (DB, `@pytest.mark.asyncio` + `db_session`, mirror `tests/services/brokers/binance/demo/test_ledger_service.py`): create account → place_limit (buy) reserves cash → reconcile with canned-crossing OHLCV → filled + position/cash/PnL correct via the existing execute path; no-cross → still pending; cancel releases reservation; sell path realized PnL. Monkeypatch the OHLCV fetch with canned bars (no live Upbit call in tests).
- **MCP tools:** DummyMCP registration + `_ORDER_SURFACE_MATRIX` DEFAULT stays green (add the name set to both places); a handler test asserting `paper_place_limit_order` dry_run default + confirm gate (mirror `tests/test_mcp_kiwoom_order_variants.py`).
- Guards: a new AST import-guard is NOT needed (no service-internal repository — reuses PaperTradingService); ROB-501 no-LLM guard passes trivially.

## Dependencies

Existing `PaperTradingService` + paper models + paper MCP tools; `market_data.service.get_ohlcv/get_quote` (crypto→Upbit); `adjust_price_to_upbit_unit`; `FEE_RATES`/`calculate_fee`; `DEFAULT_MINIMUM_VALUES`. One additive alembic migration (`paper_pending_orders`).

## Success criteria

Deposit → analysis-driven limit buy/sell (resting) → reconcile against live Upbit data → fills only when the market actually crossed, with correct derived cash/positions/realized+unrealized PnL on real Upbit symbols — end to end via MCP, interactively, so we can judge whether the analysis produces meaningful, executable crypto decisions.
