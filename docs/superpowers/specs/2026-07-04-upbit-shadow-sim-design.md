# Upbit Shadow-Sim (MCP paper-broker) — Design

Generated by brainstorming on 2026-07-04
Linear: ROB-703 (parent ROB-644, follow-up to ROB-697)
Branch: feature/ROB-703-upbit-shadow-sim
Status: DRAFT

## Problem Statement

The operator wants to validate that **meaningful crypto analysis can drive buy/sell decisions executed on a mock account** — interactively (operator + Claude feedback loop), on the currently-open crypto market. Automation/scheduling is deferred until the loop is proven.

Upbit (the operator's real crypto exchange) has **no paper/sandbox API**, and `binance_demo` is a different exchange/currency (USDT), so trading it would mean analyzing Binance coins, not the operator's Upbit portfolio — a venue disconnect. We therefore build an **Upbit shadow-sim**: a virtual (paper) KRW account whose fills are simulated against **live Upbit market data**, so the analysis and the execution are on the *same* venue (Upbit, KRW, real symbols).

### Why this over A′ (ROB-697)

ROB-697's A′ replayed the frozen `get_hermes_context` bundle and measured 0% fidelity — because that bundle is a degraded provenance/gate artifact (`buying_power=0`, market `unavailable`, stale candidates), **not** the evidence real decisions use. The real operator decisions (`~/services/auto_trader-operator/prompts/*.md`) are driven by the **full live MCP toolchain** + `route_request` procedure. The shadow-sim reuses that live analysis path; only the *execution surface* is new.

## Scope

**v1 (this spec):** the shadow-sim paper broker as MCP tools + a virtual ledger with lazy, evidence-based fills. The analysis/decision/review loop reuses the existing MCP tools and procedure (no new analysis build).

**Out of scope (fast-follow):** background fill engine, partial fills, slippage/orderbook-depth fills, auto retrospective/forecast wiring, scheduling/headless automation.

## Architecture

Four bounded units + the MCP surface, mirroring the existing mock-broker backends (`binance_demo`, `kis_mock`):

```
MCP tools (upbit_shadow_*)          app/mcp_server/tooling/orders_upbit_shadow_variants.py
        │  (thin handlers → service)
        ▼
Ledger service (only write surface)  app/services/brokers/upbit/shadow/service.py
        │
        ├── Fill engine (PURE)        app/services/brokers/upbit/shadow/fills.py
        │      dict in → fill decision out; unit-tested with canned bars/trades
        ├── Ledger repository         app/services/brokers/upbit/shadow/repository.py (service-internal)
        └── Live-data adapter         reuses existing Upbit market-data (get_quote/OHLCV/orderbook/public_trades)
        ▼
review.upbit_shadow_order_ledger      alembic migration (mirror binance_demo_order_ledger)
```

Boundary test: each unit answers "what does it do / how used / what depends on it" without reading the others' internals. The fill engine is pure so it can be exhaustively unit-tested; the repository is the only SQL; the service is the only write path; the tools are thin.

## Components

### 1. Ledger model — `review.upbit_shadow_order_ledger`

Mirror `app/models/binance_demo_order_ledger.py` structure. Columns (indicative):
- `order_id` (Text, UNIQUE — idempotency key, e.g. `ushadow-<sha16>`), `correlation_id` (Text, nullable)
- `symbol` (Text — Upbit market code, e.g. `KRW-BTC`), `side` (Text — `buy`/`sell`), `order_type` (Text — `limit`/`market`)
- `limit_price` (Numeric, nullable for market), `quantity` (Numeric, nullable), `amount_krw` (Numeric, nullable) — exactly one of quantity/amount_krw supplied
- `status` (Text CHECK — `previewed`→`placed`→`resting`→`filled`/`cancelled`; market skips `resting`)
- `fill_price` (Numeric, nullable), `fill_quantity` (Numeric, nullable), `fee_krw` (Numeric, nullable), `realized_pnl_krw` (Numeric, nullable — on sell)
- `thesis` (Text, nullable), `evidence_snapshot` (JSONB, nullable)
- `placed_at` / `filled_at` / `reconciled_at` / `cancelled_at` (timestamptz, nullable), `created_at`/`updated_at`
- `account_label` (Text, default `default`) so multiple virtual sub-accounts / resets are possible

**Virtual balance + positions are DERIVED, not stored**: `initial_deposit_krw` (a small `upbit_shadow_account` row per `account_label`) + all `filled` orders → current KRW cash and per-symbol {qty, avg_price}. This keeps a single source of truth (the ledger) and avoids balance/ledger drift.

### 2. Fill engine — pure functions (`fills.py`)

- `market_fill(side, current_price, qty_or_amount) -> Fill` — fills at `current_price` (live), applies fee.
- `limit_fill_from_evidence(side, limit_price, placed_at, bars_or_trades) -> Fill | None` — for a resting limit: buy fills if `min(low) <= limit_price` since `placed_at`; sell fills if `max(high) >= limit_price`; fill at `limit_price`; else None (still resting). No partial fills (all-or-nothing) in v1.
- `snap_to_tick(price) -> price` — Upbit KRW price-band tick table (reuse an existing tick util if present under `app/services/brokers/upbit/`; else encode the Upbit KRW tick bands). Reject sub-tick prices in preview.
- Constants: `MIN_ORDER_KRW = 5000`, `FEE_RATE = Decimal("0.0005")` (Upbit KRW maker/taker ≈ 0.05%). Verify against any existing Upbit constant.

All pure (Decimal in → Decimal/Fill out), no I/O, exhaustively unit-tested with canned bars/trades.

### 3. Live-data adapter

Reuse existing Upbit market data — **read-only**:
- current price: existing crypto `get_quote` path / Upbit client.
- OHLCV window since `placed_at`: existing crypto OHLCV reader (for limit reconcile).
- (optional) `get_orderbook(market=crypto)` for a better market-fill price.
The plan (writing-plans) must name the exact existing functions; the adapter is a thin wrapper so the service/fill-engine don't couple to the market-data source.

### 4. Ledger service (only write surface) + repository

`UpbitShadowService` — the sole writer (mirror `BinanceDemoLedgerService`); direct SQL INSERT/UPDATE forbidden (AST guard as binance_demo does). Methods: `deposit`, `preview`, `place`, `cancel`, `reconcile`, `positions`, `orderable_cash`, `reset`. `UpbitShadowRepository` is service-internal.

### 5. MCP tools — account_mode `upbit_shadow`

`app/mcp_server/tooling/orders_upbit_shadow_variants.py`, mirroring the kiwoom_mock/kis_mock registration:
- `upbit_shadow_deposit(amount_krw, account_label?)` — set/add virtual KRW (the "입금").
- `upbit_shadow_preview_order(symbol, side, order_type, limit_price?, quantity?, amount_krw?)` — validate (tick, min-notional, one-of qty/amount), project fill/cost. No write.
- `upbit_shadow_place_order(..., confirm=False)` — `dry_run`/`confirm` gate (consistency, not danger). Records `placed`; market → immediate fill; limit → `resting`.
- `upbit_shadow_cancel_order(order_id)`.
- `upbit_shadow_get_positions(account_label?)` — derived positions + KRW cash + **live-priced unrealized PnL**.
- `upbit_shadow_get_orderable_cash(account_label?)`.
- `upbit_shadow_reconcile_orders(account_label?)` — evaluate resting limits against live Upbit evidence → book fills.
- `upbit_shadow_reset(account_label?)` — clear the virtual account (for iterating).

Register on the DEFAULT profile (like other mock tools). Add `upbit_shadow` to `trade_retrospective_service._VALID_ACCOUNT_MODES` so shadow retros CAN be stored (interactive retros for v1; no auto pending-scan).

## Data flow (the interactive loop — operational, no new build)

1. `upbit_shadow_deposit(amount_krw)` — seed the virtual account.
2. Claude runs live crypto analysis: `route_request(intent="buy_analysis"|"discovery", market="crypto")` → the standard tool sequence (quotes, indicators, support/resistance, Upbit data) → proposes buy/sell with levels + thesis.
3. Operator + Claude review the proposal (feedback loop).
4. `upbit_shadow_place_order(..., confirm=True)` for agreed decisions.
5. Later `upbit_shadow_reconcile_orders()` → book fills from live evidence.
6. `upbit_shadow_get_positions()` → observe PnL. Interactively retrospect. Iterate.

## Safety

Pure simulation — **touches no real broker or Upbit** (reads live market data + writes only its own ledger). No path reaches a real order/mutation surface. `dry_run` default + `confirm` on place is for procedural consistency, not danger. **Loss-sell guard (avg×1.01) is NOT enforced** — this is a practice account and must allow stop-loss/손절 practice (precedent: kis_mock, ROB-461). ROB-501: no in-process LLM (this is deterministic broker-sim code).

## Error handling

- Preview rejects: sub-tick price, below `MIN_ORDER_KRW`, both/neither of qty/amount, unknown symbol (validate against the live-data adapter), insufficient virtual cash (buy) / insufficient position (sell) → typed errors, no ledger write.
- Reconcile with no live data (adapter fails): leave the order `resting`, return a `data_unavailable` note (never fabricate a fill).
- `place` idempotency: same `order_id` replays the stored row (no double-book), mirroring the broker ledgers.

## Testing

- **fills.py:** pure unit tests (`@pytest.mark.unit`, no DB) — market fill + fee; limit buy fills when a bar's low crosses; limit sell fills when high crosses; no-cross → None; tick-snap bands; min-notional/one-of validation.
- **service/repository:** integration tests (`@pytest.mark.integration`, DB) — deposit→place(market)→positions/cash derivation; place(limit)→resting→reconcile(canned evidence)→filled→realized PnL on sell; idempotent place; reset. Use `investment_reports_cleanup_lock`-style isolation if it touches shared tables (it uses its own table, so likely just the standard db_session).
- **MCP tools:** DummyMCP registration test (tools present on DEFAULT profile, `upbit_shadow` in the order-surface matrix if applicable) + a handler test asserting dry_run default + confirm gate.
- Guards: ROB-285-style audit (no real Upbit signed-order endpoints reachable); ROB-501 no-LLM guard.

## Open questions (resolve in writing-plans / first implementation)

1. Exact existing Upbit market-data functions for current price + OHLCV window (name them).
2. Existing Upbit KRW tick util? (reuse) — else encode the tick bands.
3. Existing Upbit fee/min-order constants? (reuse) — else define here.
4. `binance_demo_order_ledger` exact column/type conventions to mirror.
5. Whether to store `evidence_snapshot`/`thesis` on place (recommended: yes, cheap, enables later retro).

## Dependencies

Existing Upbit market-data readers; the mock-broker patterns (binance_demo ledger/service/repository, kis_mock/kiwoom tool registration); `trade_retrospective_service._VALID_ACCOUNT_MODES`. One alembic migration (the ledger table + optional account table). Migration is included in the PR but the operator runs `alembic upgrade head` on cutover.

## Success criteria

- Deposit → analysis-driven buy/sell → place → reconcile → positions with correct derived KRW cash, positions, and realized/unrealized PnL, on live Upbit symbols — end to end via MCP, interactively.
- The loop is usable in-session (Claude drives analysis + places on the sim; operator reviews) so we can judge whether the analysis produces meaningful, executable crypto decisions.
