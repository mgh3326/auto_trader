## Problem Statement

ROB-703 shipped an Upbit-faithful paper sim (resting limit orders + reconcile). The trial ran end-to-end (deposit → live crypto analysis → place → reconcile → PnL). But the sim is a **learning dead-end**: a paper decision captures almost no structured "why" and no filled paper trade ever reaches the learning corpus.

Grounding (4-reader sweep) found the learning stores all exist — `trade_retrospectives` (ROB-647 structured postmortems), `trade_forecasts` (ROB-650 resolvable probabilistic claims + Brier calibration), `analysis_artifacts` (ROB-648), `trade_journals` — but they are joined by **opt-in loose keys** and written by **manual authoring**, and the paper subsystem is disconnected from all of them:

- The ROB-703 resting-limit fill path calls `PaperTradingService.execute_order` directly and creates **no `TradeJournal`**, so paper fills are invisible to every journal-based analytic.
- `PaperPendingOrder` / `PaperTrade` have **no `correlation_id` / `artifact_uuid` / `forecast_id` / `journal_id`** — no durable link from a paper order back to the analysis and thesis that produced it.
- `build_retrospective_pending` scans **only the three live ledgers** (kis_live / live / toss_live); it never scans paper, so a filled paper trade never surfaces as a pending retrospective.
- `_VALID_ACCOUNT_MODES` has kis_mock / kiwoom_mock / alpaca_paper but **no `paper` mode** for the paper_trading subsystem.
- `forecast_save` has zero callers outside the MCP boundary and `trade_forecasts` has no account_mode discriminator — a paper decision can't attach or calibrate a forecast today.

The goal: wire the paper sim into a complete **analysis → decision → outcome → retrospective + calibration** loop, so mock trading accumulates queryable, structured learning. Do it mock-first (safe proving ground), with a `correlation_id` spine + paper/real discriminator that extends unchanged to the real account later (step 2, out of scope here).

## What Makes This Cool

Two things that only work because it's mock:

1. **The mock loop is the ONLY place that can accumulate stop-loss outcome data.** The operator deliberately blocked loss-selling in the real code path ("매도는 내가 손절 매도를 코드 단계에서 막아뒀어" — the avg×1.01 loss-sell guard, fail-closed). So the real account structurally cannot produce "I cut a loss and here's what happened" data. The paper sim already allows loss-sells (ROB-703 deliberately left the guard off; ROB-461 precedent). Wiring stop-loss sells as a first-class retrospective/forecast category means the operator can finally measure a question they can't answer any other way: **does cutting losses actually help net PnL, or would holding have recovered?**

2. **A deterministic `correlation_id` spine makes the decision→outcome corpus self-assemble.** Today the corpus is scattered across five table groups joined by inconsistently-populated nullable keys. A stable id minted at place-time and carried order → fill → journal → forecast → retrospective turns "did my thesis pay off?" from a manual join heuristic into a real query.

## Constraints

- **Decisions this session:** D1 = mock loop first (real is step 2). D2 = loss-sells captured as a first-class learning category; a resting sell-STOP order type (sell-if-price-drops) is a **fast-follow, not v1**. D3 = Approach B (provenance spine).
- Reuse existing stores (retrospective / forecast / artifact / journal); minimize new tables. One additive migration for the `correlation_id` + linkage columns on the paper tables.
- No real-money wiring in v1. The spine is designed so real is a later, separate step.
- **Retrospective lesson stays authored** (human + Claude in-session), not auto-drafted from a heuristic. Automation captures provenance and surfaces outcomes; judgment (lesson / root-cause / next-action) is written.
- ROB-501: no in-process LLM in `app/**` (this is deterministic capture + read code).
- Paper is pure sim — no real broker / Upbit mutation; reads live market data + writes paper tables only.

## Premises (agreed 2026-07-05)

1. **Stores already exist — this is a wiring + provenance problem, not a new-store problem.** Reuse retrospective / forecast / artifact / journal; add minimal columns, not new subsystems.
2. **The keystone is decision provenance.** Paper decisions capture no structured "why" (no journal, correlation_id, forecast, target/stop). Without it a retrospective has nothing to cite. Capturing provenance at place-time is the keystone.
3. **Mock = proving ground; the spine extends to real unchanged.** No real-money wiring yet. Same `correlation_id` spine + a paper/real discriminator so aggregates separate mock from real.
4. **Automation boundary = provenance auto, lesson authored.** Place-time deterministically captures the decision's "why" (thesis / target / stop / forecast) and fills auto-surface as pending retrospectives; the lesson / root-cause / next-action is authored in a session (heuristic auto-lessons are noise).

## Approaches Considered

### Approach A: Minimal — surface-only
Add `correlation_id` + a `paper` account_mode; extend `build_retrospective_pending` to scan filled paper trades and surface them as pending retrospectives; author manually (as with real).
- Effort S–M, Risk Low. Reuses the whole retrospective pipeline.
- Rejected: leaves the keystone gap (premise 2). The decision "why" is still lost at place-time — no journal / forecast / target / stop — so a retrospective can only cite thesis text + outcome. Delivers half of "accumulate meaningful data."

### Approach B: Ideal — provenance spine (RECOMMENDED)
Mint a deterministic `correlation_id` at place-time; carry it PaperPendingOrder → PaperTrade. Capture structured provenance at place-time (target / stop / strategy / entry snapshot) by reusing the paper `TradeJournal` path + linking an `analysis_artifact`. Optionally `forecast_save` at place-time (target / probability), linked by `correlation_id`, so calibration works for paper. Extend `build_retrospective_pending` to scan paper fills (keyed by `correlation_id`). Add a `paper` account_mode discriminator. Make `stop_loss` a first-class `trigger_type`. Lesson authored in-session, now with full provenance to cite.
- Effort M–L, Risk Med. Reuses retrospective / forecast / artifact / journal stores.

### Approach C: Creative — event-driven auto-draft + unified corpus view
Approach B plus: reconcile auto-drafts a retrospective skeleton at fill (outcome / realized_pnl / trigger auto-filled, lesson blank), auto-resolves the linked forecast, and adds a `get_decision_corpus` join view.
- Effort L–XL, Risk Med-High.
- Rejected for v1: the auto-draft skeleton conflicts with premise 4 (lesson = authored); the unified view is a genuine follow-on but widens scope past the chosen wedge.

## Recommended Approach

**Approach B — provenance spine.** It is the keystone (premise 2): the missing piece is decision provenance, and B captures it at place-time while reusing every existing store. It realizes the full mock loop (analysis → decision → outcome → retrospective + calibration), and the `correlation_id` spine + paper/real discriminator is exactly the part that extends to the real account in step 2 (the D1 sequencing). A is too thin (provenance gap remains); C over-reaches and breaks the authored-lesson premise.

**Shape of the spine (for the eng plan to detail):**
- **Place-time:** `paper_place_limit_order` mints a deterministic `correlation_id` = `paper:{account_id}:{sha16(canonical)}` where `canonical` **MUST include the KST trade-day and a rung/sequence discriminator** (mirror ROB-653's P6-B idempotency key: canonical decision fields | KST trade-day | rung). Without the trade-day + rung salt, a re-placed order after cancel — or two identical ladder rungs — collide on one `correlation_id`, and because `TradeRetrospective.correlation_id` is UNIQUE and coverage dedups on it, one retrospective would silently "cover" two distinct orders. Writes the id on `PaperPendingOrder`, records a structured paper `TradeJournal` (thesis / strategy / target / stop / entry price+indicator snapshot) linked by `correlation_id`, optionally links an `analysis_artifact`. **Forecast (in v1):** `forecast_save` with a `price_target` kind carrying the same `correlation_id`, so it auto-resolves against daily OHLCV via the existing `resolution_source='ohlcv_day'` path (no new resolver). `probability` is **caller-supplied** by the MCP/Claude session, not invented by the deterministic place-time code (ROB-501 forbids in-process LLM).
- **Fill (reconcile):** carry `correlation_id` PaperPendingOrder → PaperTrade; ensure the fill produces / updates the journal with realized_pnl (the ROB-703 path currently bypasses the journal — this is the specific bridge to build). Paper orders are all-or-nothing (`PaperPendingOrder.status` has no `partially_filled`; reconcile fills full qty), so no partial-fill retrospective semantics are needed; cancelled orders produce no `PaperTrade` and so self-exclude from a fills-only surface.
- **Surface:** `build_retrospective_pending` gains a paper source (filled `PaperTrade` rows not yet covered), `suggested_correlation_id` = the stored spine id (NOT the live-ledger `{ledger}:{ref}` form — a new source branch, no precedent since even `alpaca_paper` isn't scanned today). It suggests `trigger_type=stop_loss` when `side='sell' AND realized_pnl < 0` (the avg-buy price is gone — `PaperPosition` is deleted on full close — but `PaperTrade.realized_pnl` persists). This is an **authoring suggestion**; `trigger_type` is set by the human/Claude on the retrospective, not auto-written.
- **Segment:** add `paper` to `_VALID_ACCOUNT_MODES` + the retrospective model CHECK (a *named* constraint → migration DROP+ADD, not ALTER). Paper-vs-real forecast split needs **no new column on `trade_forecasts`** — it derives from the `correlation_id` `paper:` prefix. `get_retrospective_aggregate` already filters/`group_by`s `account_mode`, so mock-vs-real + `trigger_type=stop_loss` become queryable.
- **Author:** the operator + Claude write the retrospective in-session, citing the now-complete provenance.

## Open Questions

1. **Journal bridge:** the ROB-703 resting-limit fill calls `execute_order` directly and skips the `_place_paper_order` journal path. Do we route the resting-limit fill through a shared journal-writing booker, or teach `reconcile_pending_orders` to write the journal? (Eng-review call.)
2. **Forecast resolution (decided):** v1 uses the existing `price_target` kind resolved against daily OHLCV (`resolution_source='ohlcv_day'`, `resolve_forecast`) — no new resolver, already works for crypto. A realized-PnL-tied "did my entry thesis pay off" forecast (a resolution_source reading the paper ledger) is a **fast-follow**, not v1.
3. **Account_mode value (recommended: `paper`):** the paper_trading subsystem is multi-market (equity_kr/us/crypto), so `upbit_paper` is too narrow; `paper` matches the subsystem name. Pin `paper` before migration #1 executes (it gates the CHECK-constraint value).
4. **STOP order type (D2 fast-follow):** resting sell-STOP (fill when low ≤ stop) is deferred; confirm it stays out of v1.
5. **Profile split (carried from ROB-703):** paper-limit tools live on the DEFAULT profile, `create_paper_account`/performance on DB_PAPER — the loop spans two profiles. Fold a consolidation into this work or keep separate?

## Success Criteria

- A filled paper trade automatically appears in `trade_retrospective_pending`, keyed by a `correlation_id` (trade-day + rung salted) that also links its journal + forecast + (optional) analysis_artifact.
- Pre-existing rows from the ROB-703 trial (account `shadow-trial-20260704`) surface as **thin/legacy** retrospectables (synthesized `correlation_id` + outcome from the stored `realized_pnl`; provenance columns backfill NULL). A **re-run** of the trial under the new place-time capture (Next Steps #5) produces the first **full-provenance** mock retrospective.
- `get_retrospective_aggregate` can `group_by` account_mode (paper vs real, paper derived from the `correlation_id` prefix) and `trigger_type=stop_loss` — i.e. the operator can query "my mock stop-loss win-rate / net PnL."
- A paper decision carries a `price_target` forecast (same `correlation_id`) that `forecast_resolve` auto-scores via `ohlcv_day`, producing a Brier calibration number for mock decisions.
- Nothing in the paper path touches a real broker; existing real-account behavior unchanged.
